# Changeset 日志

`changeset` 用来记录每一次产品、策略、参数、评测或基础设施改变，目标是让系统后续能够回答：

- 这次为什么改？
- 改了哪些功能或策略口径？
- 预期改善什么指标？
- 实际验证结果是什么？
- 如果变差，怎么回滚？

它和「工作总结」配合使用：

1. `changeset` 记录每次改变的意图、范围和预期。
2. 「工作总结」持续记录系统评分、收益、风险、策略健康度和历史快照。
3. 每周复盘时，把 changeset 的预期和工作总结的实际变化对照，决定继续优化、回滚或观察。

## 命名

文件放在 `docs/changesets/`，命名格式：

```text
YYYY-MM-DD-short-title.md
```

示例：

```text
2026-07-01-work-summary-center.md
2026-07-08-etf-rotation-risk-control.md
```

## 类型

- `feature`：新增功能或页面
- `strategy`：策略规则、参数、仓位或风控口径变化
- `eval`：评测集、harness、评分方法变化
- `data`：数据源、存储、采集、清洗变化
- `ops`：定时任务、部署、通知、运行机制变化
- `fix`：问题修复
- `docs`：文档和说明变化

## 状态

- `draft`：计划中，尚未实现
- `shipped`：已实现并通过基本验证
- `observing`：已上线观察，等待更多样本
- `validated`：已通过一段时间数据确认有效
- `reverted`：已回滚或废弃

## 每次改动都要写什么

复制 [template.md](./template.md)，至少填写：

- 背景与问题
- 改动范围
- 预期影响
- 验证方式
- 后续观察指标
- 回滚方案

如果改动会影响收益、风险、策略默认参数、自动执行或用户决策，必须写 changeset。
